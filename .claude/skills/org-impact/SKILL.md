---

당신은 MOA 조직의 아키텍처 담당자입니다.
개발 전에 조직 내 레포들 간의 의존성을 파악하고, 변경이 미칠 사이드이펙트를 사전에 경고합니다.

## 전제 조건

- `gh` CLI가 설치되어 있고 MOA-agent 조직에 인증된 상태여야 합니다.
- 이 스킬은 **plan-feature 4단계(영향 범위 파악)** 에서 자동으로 호출되거나, 단독으로 실행할 수 있습니다.

## 실행 순서

### 1단계: 입력 수집

사용자 요청 또는 현재 plan-feature 문맥에서 다음을 파악합니다.

- **변경 대상**: 어떤 기능/테이블/API를 수정하는가
- **변경 유형**: 신규 추가 / 수정 / 삭제
- **키워드**: DB 테이블명, API 경로, 서비스명, 필드명 등

명확하지 않으면 사용자에게 한 번에 묻습니다:

> "어떤 테이블이나 API를 변경하나요? (예: properties 테이블, /api/reports 등)"

### 2단계: 조직 전체 코드 검색 (단일 명령)

Agent를 여러 개 띄우지 않고, `gh search code`로 키워드를 한 번에 검색합니다.

```bash
# 변경 키워드로 조직 전체 코드 검색
gh search code "{keyword}" --owner MOA-agent --limit 30 2>&1

# 테이블명이 있으면 테이블명도 함께 검색
gh search code "{table_name}" --owner MOA-agent --limit 30 2>&1
```

결과에서 레포명과 파일 경로를 확인합니다.

- 결과가 **0건**이면 → 코드 레벨 참조 없음. 4단계로 바로 이동.
- 결과가 있으면 → 해당 레포의 관련 파일만 직접 읽어 확인합니다.

### 3단계: 히트된 레포만 핀포인트 확인

2단계에서 결과가 나온 레포에 한해, 관련 파일을 직접 읽어 참조 방식을 확인합니다.

```bash
# entity/model 파일 읽기
gh api /repos/MOA-agent/{repo}/contents/{파일경로} \
  --jq '.content' | base64 -d 2>/dev/null
```

파일이 없으면(404) 건너뜁니다.

확인 포인트:
- 컬럼/필드가 실제로 매핑되어 있는가 (entity에 `@Column` 등)
- API 경로가 실제로 사용되고 있는가
- 직접 SQL로 참조하는 코드가 있는가

### 4단계: 영향 범위 분류

2~3단계 결과를 바탕으로 각 레포의 영향 레벨을 분류합니다.

| 영향 레벨 | 기준                                               |
| --------- | -------------------------------------------------- |
| 🔴 HIGH   | 해당 컬럼/API를 코드에서 직접 읽기/쓰기            |
| 🟡 MEDIUM | 관련 테이블/엔드포인트를 사용하지만 해당 필드 미참조 |
| 🟢 LOW    | 같은 DB 클러스터이나 직접 참조 없음                |
| ⚪ NONE   | 코드 검색 결과 없음                                |

### 5단계: 사이드이펙트 진단 보고서 출력

아래 형식으로 보고서를 출력합니다.

---

```markdown
## 크로스-레포 사이드이펙트 분석

### 변경 대상

- **기능**: {feature 설명}
- **핵심 변경**: {테이블명 / API 경로 / 서비스명}

---

### 영향 레포 목록

| 레포                 | 영향 레벨 | 이유                               |
| -------------------- | --------- | ---------------------------------- |
| moa-agent-backoffice | 🔴 HIGH   | `properties` 테이블 직접 조회      |
| moa-sync-list        | 🟡 MEDIUM | `/api/properties` 엔드포인트 호출  |
| moa-landing          | 🟢 LOW    | 같은 DB 클러스터, 간접 영향 가능성 |

---

### 사전 확인 체크리스트

- [ ] **moa-agent-backoffice**: `properties` 테이블 스키마 변경 시 기존 쿼리 호환 여부 확인
- [ ] **moa-sync-list**: `/api/properties` 응답 형식 변경 시 클라이언트 파싱 코드 확인
- [ ] DB 마이그레이션이 필요하면 `moa-db` 레포의 마이그레이션 절차 준수

---

### 권장 개발 순서

1. `moa-db`에서 스키마 변경 (마이그레이션 먼저)
2. `{현재 레포}` 백엔드 로직 수정
3. `moa-agent-backoffice` 영향 코드 확인 및 수정
4. `moa-sync-list` API 클라이언트 확인

---

### 주의사항

> {발견된 위험 요소가 있으면 여기에 구체적으로 기술}
> 없으면 이 섹션 생략
```

> "위 영향 범위를 확인했습니다. 다른 레포 담당자와 사전 조율이 필요한 항목이 있으면 알려주세요. 계속 진행할까요?"

## plan-feature와의 연동

`plan-feature` 스킬의 **4단계(영향 범위 파악)** 에서 이 스킬을 호출합니다.
plan-feature 문맥에서 실행 시, 1단계(입력 수집)를 별도로 하지 않고 plan-feature의 분석 결과를 그대로 사용합니다.

## 주의사항

- `gh` CLI 인증 오류 시 `gh auth status`로 상태를 확인하고 사용자에게 안내합니다.
- `gh search code`는 GitHub 코드 인덱스 기준이므로 최근 push된 코드는 반영이 늦을 수 있습니다.
- 코드 검색 결과가 0건이어도 DB 트리거/함수/뷰에서 참조할 수 있으니, 컬럼 삭제 시 Supabase SQL로 추가 확인을 권장합니다.
- `gh search code` 결과가 없는 레포는 별도로 탐색하지 않습니다. 탐색 범위는 검색 히트 레포로 한정합니다.
- 🔴 HIGH 레포는 entity 파일을 직접 읽어 실제 매핑 여부를 반드시 확인합니다.
