당신은 MOA 팀의 코드베이스를 잘 아는 전문가입니다.
현재 작업 디렉토리의 레포를 직접 탐색해서 `README.md`를 작성합니다.

## 목표

모아나(Slack 위키봇)가 이 레포를 빠르고 정확하게 이해할 수 있도록 구조화된 `README.md`를 생성합니다.
모아나는 레포에 진입하면 README.md를 가장 먼저 읽습니다. 이 파일이 잘 쓰여있으면 불필요한 파일 탐색 없이 바로 핵심 파일로 이동합니다.

---

## 실행 순서

### 1단계: 레포 기본 파악

**docs/ 문서를 먼저 읽으세요.** 아래 파일이 존재하면 코드 탐색 전에 읽고, 각 파일에서 파악된 항목은 2단계에서 건너뜁니다.

| 파일 | 활용 항목 |
|---|---|
| `docs/project-overview.md` | 한 줄 요약, 책임 영역, 주요 기능 목록 |
| `docs/infra.md` | 기술 스택, 환경변수 목록, 외부 서비스 연동 |
| `docs/deploy.md` | 개발 시작 명령어, 배포 방식 |

docs/ 파일이 없으면 아래 순서로 직접 파악합니다:

1. `package.json` 또는 `pyproject.toml` 또는 `build.gradle` — 프로젝트 이름, 의존성, 스크립트
2. `.env.example` 또는 `.env` (값은 무시, 키 이름만) — 환경변수 목록
3. 루트 디렉토리 목록 — 전체 구조 파악
4. `src/` 또는 `app/` 하위 디렉토리 목록 — 주요 모듈 파악
5. 기존 `README.md`가 있으면 읽고 내용 참고 (단, 그대로 복사하지 않음)
6. `CLAUDE.md`가 있으면 읽기 — 팀 내부 컨텍스트 확인

### 2단계: 핵심 파일 탐색

아래 우선순위로 탐색하세요. **모든 파일을 읽을 필요는 없습니다.** 레포 역할을 파악하는 데 필요한 것만 선택적으로 읽으세요. **1단계 docs/에서 이미 파악된 항목은 건너뜁니다.**

**진입점 파악 (반드시)**
- `src/index.ts`, `src/main.ts`, `src/app.ts`, `src/server.ts`, `main.py`, `app.py` 등
- Next.js라면 `src/app/` 구조, NestJS라면 `src/app.module.ts`

**도메인 모듈 파악 (레포별 핵심)**
- `src/modules/`, `src/features/`, `src/domain/` 하위 디렉토리 목록
- 각 모듈의 `*.controller.ts`, `*.service.ts`, `*.router.ts`, `*.handler.ts` 파일명 확인 (내용 전체 읽기 불필요 — 파일명만으로 도메인 추론)

**외부 API 호출 탐색 (docs/infra.md가 없을 때만)**
- `src/lib/api/`, `src/api/`, `src/services/`, `src/clients/` 등 API 클라이언트 위치 확인
- 이 디렉토리의 파일명과 내용 일부를 읽어 어떤 외부 서비스/레포를 호출하는지 파악
- `fetch(`, `axios.`, `HttpService`, `http.get` 같은 패턴 검색

**환경변수 → 의존성 단서 (docs/infra.md가 없을 때만)**
- `API_URL`, `BACKEND_URL`, `SERVICE_URL` 같은 변수명이 있으면 다른 레포에 의존함을 의미

### 3단계: 분석 및 판단

탐색한 내용을 바탕으로 다음을 결정하세요:

**한 줄 요약** — "무엇을 + 어떤 기술로" 한 문장. 예:
- ✅ "MOA 사무실 매물 검색·채팅을 담당하는 Next.js 15 앱"
- ✅ "매물·임차인·중개인 데이터를 관리하는 NestJS REST API 서버"
- ❌ "This is a Next.js project" (역할 불명확)

**책임 영역** — 기능/도메인 단위로 묶기. 파일 하나가 아닌 디렉토리 경로로 표현. 모르면 추측하지 말고 파악 가능한 것만 작성.

**레포 간 의존성** — 외부 API 호출이 있으면 어느 레포인지 추론. MOA 레포 목록:
- `moa-next` — 프론트엔드 (Next.js)
- `moa-backoffice` — 백오피스 (Next.js)
- `moa-nest` — 메인 백엔드 API (NestJS)
- `moa-db` — DB 스키마 관리
- `moa-lms` — LMS
- `moa-partners` — 파트너 포털
- `moa-sync-list` — 데이터 동기화
- `moa-landing` — 랜딩 페이지
- `moana` — Slack 위키봇 (assistant/moana)

**파일 맵** — 핵심 파일 5~7개만 선택. 역할 설명 없는 파일은 포함하지 않음.

**핵심 플로우** — 주요 기능 2~3개. 코드에서 확인된 실제 흐름만. 추측으로 채우지 않음.

**도메인 용어** — 코드에서 발견한 팀 내부 용어, 이 레포에서만 쓰이는 의미가 있는 것만.

### 4단계: README.md 작성

아래 형식을 **정확히** 따라 작성합니다.
확인되지 않은 내용은 추측하지 말고 해당 항목을 생략하거나 `—`로 표시하세요.

```markdown

> [한 줄 요약]

**이 레포의 책임 영역**

| 도메인 | 위치 |
|--------|------|
| [도메인] | `[경로]` |

**레포 간 의존성**

| 방향 | 대상 레포 | 사용 영역 | 이 레포의 진입점 |
|------|----------|----------|----------------|
| 사용함 | `[레포명]` | [영역] | `[파일경로]` |
| 사용됨 | `[레포명]` | [영역] | — |

(의존성이 없으면 이 섹션 생략)

**핵심 도메인 용어**

| 용어 | 설명 |
|------|------|
| [용어] | [이 레포 안에서의 의미] |

### 5단계: 확인 후 저장

README.md 전체 내용을 채팅창에 먼저 보여주세요.
그 다음 사용자에게 확인을 요청하세요:

> "위 내용으로 `README.md`를 저장할까요? 수정이 필요한 부분이 있으면 알려주세요."

사용자가 확인하면 `README.md` 파일을 프로젝트 루트에 저장합니다.
이미 `README.md`가 있으면 덮어쓰기 전에 경고합니다.

## 작성 원칙

**써야 하는 것**
- 이 레포만 알 수 있는 것 (내부 도메인, 파일 역할, 의존 관계)
- 코드만 봐서는 파악하기 어려운 맥락
- 팀 내부 용어·약어

**쓰지 말아야 하는 것**
- 코드 보면 알 수 있는 것 (함수 목록, 변수 설명)
- `npm install`, `bun install` 같은 당연한 명령어 나열
- "TODO", "추후 업데이트 예정" 같은 미완성 내용
- 추측에 기반한 내용 — 확인된 것만 작성
