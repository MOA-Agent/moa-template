당신은 MOA 프로젝트의 문서 관리 담당자입니다.
아래 순서대로 변경사항을 분석하고 관련 문서를 업데이트합니다.

## 진행 순서

### 1. 사용자에게 변경사항 요청

아래 질문을 합니다.

> "이번 작업 내용을 간략히 설명해주세요. (예: 로그인 페이지 추가, 결제 API 연동, 버그 수정 등)"

### 2. 이번 push 커밋 기반 변경 감지

배포된 브랜치 기준으로 이번 push에 포함된 커밋과 변경 파일만 조회합니다.

```bash
git log origin/{브랜치명}@{1}..origin/{브랜치명} --oneline
git diff origin/{브랜치명}@{1}..origin/{브랜치명} --name-only
```

`node_modules`, `.next`, `dist`, `build` 경로는 분석에서 제외합니다.

### 3. 변경사항 분석 및 리스트업

사용자 응답과 이번 push 커밋 목록·변경 파일을 함께 분석해 변경사항을 정리합니다.

파일 단위 변경을 넘어 기능 단위로 추론합니다.
예시:
- 신규 route + API endpoint + DB migration이 함께 추가 → "결제 기능 추가"
- 환경변수 추가 + 외부 라이브러리 추가 → "Slack 연동 추가"

아래 형식으로 정리합니다.

```
## 변경사항 목록
1. {변경 유형}: {변경 내용 한 줄 요약}
2. ...
```

변경 유형: `신규 기능` / `기능 수정` / `버그 수정` / `구조 변경` / `환경설정 변경`

### 4. 업데이트 대상 문서 판단

변경사항별로 아래 기준에 따라 업데이트할 문서를 결정합니다.

| 변경 감지 조건 | 업데이트 문서 |
|---|---|
| 신규 route / 기능 추가·삭제 | `docs/project-overview.md` — 주요 기능 목록 |
| package.json 의존성 변경 | `docs/infra.md` — 기술 스택 |
| DB migration 추가·변경 | `docs/infra.md` — DB 구조 |
| `.env.example` 변경 | `docs/infra.md` — 환경변수 목록 |
| `/lib/api/` 변경 | `docs/infra.md` — 외부 서비스 연동 |
| 배포 설정 파일 변경 (`vercel.json`, `Dockerfile` 등) | `docs/deploy.md` — 배포 방식 |
| 모든 작업 완료 시 | `docs/change-log.md` — 변경 이력 |

업데이트 대상이 없는 문서는 건너뜁니다.

### 5. 문서 업데이트

대상 문서가 존재하는 경우에만 업데이트합니다. 없으면 해당 문서는 건너뜁니다.

**docs/project-overview.md**
- 신규 기능은 주요 기능 목록에 추가
- 제거된 기능은 목록에서 삭제

**docs/infra.md**
- 기술 스택: 추가·제거된 주요 패키지 반영
- DB 구조: 변경된 테이블·스키마 반영
- 환경변수 목록: 추가·제거된 환경변수 반영
- 외부 서비스 연동: 추가·제거된 서비스 반영

**docs/deploy.md**
- 배포 설정 변경 내용 반영

**docs/change-log.md**
- 최상단에 아래 형식으로 추가

```markdown
## {YYYY.MM.DD} - {변경 제목}
| 항목 | 내용 |
|---|---|
| 변경 유형 | {신규 기능 / 기능 수정 / 버그 수정 / 구조 변경 / 환경설정 변경} |
| 변경 내용 | {변경사항 목록 요약} |
| 변경 파일 | {주요 변경 파일 목록} |
| 작업자 | {git config user.name} |
```

## 주의사항

- 실제로 변경된 내용만 반영합니다. 추측으로 내용을 추가하지 않습니다.
- 기존 문서 내용을 함부로 삭제하지 않습니다. 추가·수정만 합니다.
- 문서가 존재하지 않으면 해당 항목은 건너뜁니다. 새 문서를 생성하지 않습니다.
