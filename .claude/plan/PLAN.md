### **1. 템플릿의 역할**

<aside>

- 신규 프로젝트 진행 시, 내부 기준에 맞는 개발 환경을 쉽게 구성할 수 있어야 함.
- 기능 개발 시, 디자인 시스템 / 코드 컨벤션 / 보안 기준을 일관되게 적용할 수 있어야 함.
- 프로젝트 구조 및 설정 정보를 기반으로 내부 문서가 자동 생성·업데이트될 수 있어야 함.
- 신규 담당자도 프로젝트 구조를 빠르게 이해할 수 있어야 함.
</aside>

### **2. 템플릿 기본 구조**

**템플릿 적용 범위**

> 웹 기반, Vercel/EC2 배포, Next.js 사용 프로젝트를 우선 대상으로 함. 
이외 유형은 필요 시 진행
> 
- **초기 적용 대상**
    - Next.js 기반 웹 프로젝트
    - Vercel / EC2 배포 (단, EC2 배포는 직접 설정)
    - Supabase / Rds 사용 (단, RDS는 직접 설정)
- **제외 범위**
    - 데이터 파이프라인 프로젝트
    - AI 실험 서버

**템플릿 구조**

### 3. Skill 동작 방식

**3.1. init skill**

---

- 어떤 유형의 프로젝트를 만들 것인지 사용자에게 질문하고, 사용자의 응답에 맞는 프로젝트 구조를 세팅하는 Skill

<aside>

**Skill 동작 방식**

1. 질문 (L1 → L2 → L3 순서로 진행)
2. 선택 조합 확인
3. 코드/설정 파일 생성
4. 문서 파일 생성
</aside>

**3.1.1. 질문 설계**

`L1. 프로젝트 기본 질문`

1. 프로젝트 이름은?
2. 프로젝트 한 줄 설명은?
3. 왜 만들었나요? / 어떤 문제를 해결하나요? (프로젝트 목적)
4. 주요 사용자는 누구인가요? (예: 내부 직원 / 외부 고객 / 특정 팀)
5. 주요 기능을 간단히 나열해주세요. (예: 로그인, 대시보드, 리포트 다운로드)

        ## 배포하는 방법 - init skill

        > 배포는 Claude에게 요청하면 됩니다. Claude가 보안 점검 → 배포 → 결과 확인까지 자동으로 진행합니다.

        ### 개발 환경 배포
        1. Claude Code를 열고 이 프로젝트 폴더에서 시작합니다.
        2. 채팅창에 아래 중 하나를 입력합니다:
           ```
           개발 배포해줘
           ```
        3. Claude가 배포 전 체크리스트를 확인하고, 문제가 없으면 배포를 진행합니다.
        4. 완료되면 개발 환경 URL을 알려줍니다.

        ### 운영 환경 배포
        1. Claude Code를 열고 이 프로젝트 폴더에서 시작합니다.
        2. 채팅창에 아래 중 하나를 입력합니다:
           ```
           운영 배포해줘
           ```
        3. Claude가 보안 점검을 실행합니다. 문제가 발견되면 배포가 중단되고 조치 방법을 안내합니다.
        4. 모든 점검을 통과하면 배포를 진행하고, 완료되면 운영 URL을 알려줍니다.

        > **배포가 중단된 경우:** Claude가 안내한 문제를 해결한 뒤 다시 배포를 요청하면 됩니다.

**3.2. complete-task skill**

---

- 신규 개발 작업 및 수정 작업 등을 완료했을 때, 관리되어야 하는 프로젝트 문서를 자동으로 업데이트해주는 skill

<aside>

**skill 동작 방식**

1. 수동 질문 (작업자에게 직접 확인이 필요한 내용)
2. git diff 변경 감지 
3. 변경사항 리스트업 (무엇이 바뀌었는가) 
4. 변경사항 기준 업데이트 대상 판단 (어떤 문서를 업데이트할 것인가)
5. 문서 업데이트 (감지 결과 + 질문 응답 기반으로 각 문서에 무엇을 업데이트할 것인가)
</aside>

**3.2.1. 변경 감지**

- Git Diff 기반으로 변경된 파일 및 내용을 감지
- 작업 완료 시점에 skill 실행, 마지막 커밋 기준으로 변경 범위 파악
- Git Diff 기반으로 변경된 파일 및 내용을 감지한다.
- 파일 단위 감지 외에, 여러 변경을 종합해 기능 단위로 추론한다. 예를 들어 신규 route + API endpoint + DB migration이 함께 추가된 경우, "결제 기능 추가됨"과 같이 맥락을 파악하여 목록에 포함할 수 있다.
- 감지 결과는 수동 질문 응답과 함께 번호가 붙은 목록으로 정리되며, 이후 문서 업데이트의 기준이 된다.

**피드백**

---

**1. Hook → Skill 전환**

Claude Code의 Hook은 자동 트리거 방식이라 의도한 동작 방식과 맞지 않음. Claude가 액션할 때 맥락에 맞는 Skill을 참고해서 실행하는 방식으로 전환.

**개선 방향:** 전체 Hook 구조를 Skill 기반으로 재설계. CLAUDE.md에서 언제 어떤 Skill을 실행할지 트리거 조건 정의.

---

**2. share-update skill 공유 자동화**

문서 생성 후 공유 과정이 수동이면 누락 가능성 있음. 생성부터 공유까지 한 번에 처리되어야 함.

> **개선 방향:** share-update skill 실행 시 HTML 생성 → main 브랜치 업로드 → 슬랙 특정 채널 자동 공유까지 한 번에 처리.
> 

---

**3. 내부 규칙 Skill 분리**

디자인 시스템 / 코드 컨벤션 등 내부 규칙을 CLAUDE.md에 직접 작성하면 관리가 어려워짐.

> **개선 방향:** 내부 규칙은 별도 Skill 파일로 분리 (`MOA_DESIGN.md` 등). CLAUDE.md에서는 동작 방식과 트리거 조건만 정의.
> 

---

**4. 사용자 시나리오 기반 Skill 설계**

개발 규칙을 모르는 작업자도 원하는 작업을 올바른 루트로 진행할 수 있어야 함.

> **개선 방향:** 주요 사용자 시나리오 리스트업 후 각 시나리오별 대응 Skill 정의. 예외 상황 처리 방식도 함께 명세. `develop.md` (진입점) → `plan-feature.md` → `build-feature.md` 구조로 설계.
> 

---

**5. README.md 위키봇 가이드**

위키봇이 프로젝트를 파악할 때 어떤 문서를 참고해야 하는지 안내가 필요함.

> **개선 방향:** init skill 실행 시 README.md 자동 생성. 문서 목록 / 각 문서 설명 / 참고 시점 / Skill 목록 포함.
> 

---

**6. EC2 / RDS 인프라 설정 범위 제외**

EC2 / RDS 세부 설정은 인프라 담당자 영역으로, 템플릿 자동화 범위에 포함하기 어려움.

> **개선 방향:** EC2 / RDS 관련 세부 인프라 설정은 템플릿 자동화 범위에서 제외. 담당자가 직접 설정하도록 안내 문구만 포함.
